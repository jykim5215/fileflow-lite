# FileFlow Lite — 프로젝트 인계 문서

## 1. 프로젝트 개요와 목표

FileFlow Lite는 Windows 파일 탐색기에서 번거로운 파일 정리 작업을 안전하게 보완하는 무료·오픈소스 데스크톱 유틸리티다. 로컬 서버나 유료 서비스 없이 장치 안에서 바로 실행하며, 배포판은 가능한 한 단일 `.exe`와 휴대용 `.zip` 형태로 제공한다.

핵심 목표는 다음과 같다.

- 하위 폴더의 파일을 한 폴더로 안전하게 모으는 **폴더 평탄화**
- 여러 파일을 규칙에 따라 `1, 2, 3...` 순서로 바꾸는 **일괄 순번 이름짓기**
- 모든 실제 변경 전에 **미리보기와 명시적 사용자 확인** 제공
- 직전 작업의 **실행 취소(Undo)** 및 감사 가능한 로컬 로그 제공
- Windows 탐색기 우클릭 메뉴에서 주요 화면으로 바로 진입
- GitHub Releases를 이용한 무료 배포와 사용자 승인 기반 업데이트 확인

작업명은 `FileFlow Lite`이며, 사용자가 원하는 이름이 있으면 구현 전에 변경할 수 있다.

## 2. 사용자 요구사항 정리

### 2.1 폴더 평탄화

- 사용자가 선택한 원본 폴더의 모든 하위 폴더를 재귀 순회한다.
- 발견한 파일을 사용자가 지정한 대상 폴더 한 곳으로 복사하거나 이동한다.
- 기본 동작은 원본을 보존하는 `복사`다.
- 이름 충돌 정책을 선택할 수 있다.
  - 순번 자동 부여: `photo.jpg`, `photo (2).jpg`, `photo (3).jpg`
  - 원래 폴더명 접두어: `여행_photo.jpg`; 그래도 충돌하면 안전 순번을 추가
- 이동 작업 후 빈 폴더를 삭제할지 선택할 수 있다. 복사 모드에서는 원본 폴더를 삭제하지 않는다.
- 원본과 대상이 같거나 대상이 원본의 하위 폴더인 경우, 재귀 자기복사와 범위 이탈을 방지하도록 차단하거나 명확히 제외한다.

### 2.2 일괄 순번 이름짓기

- 선택한 파일을 정렬한 뒤 연속 번호로 이름을 변경한다.
- 접두어, 접미어, 번호 자릿수 패딩, 시작 번호를 설정할 수 있다.
- 정렬 기준은 이름, 수정일, 크기를 제공하고 각 기준의 오름차순/내림차순을 지원한다.
- 파일 확장자는 원본 그대로 보존한다.
- 같은 폴더 안에서 이름을 서로 교환하는 경우까지 안전하도록 임시 고유 이름을 거치는 2단계 변경을 사용한다.

### 2.3 필수 안전장치

- 두 기능 모두 `검사 → 미리보기 → 사용자 확인 → 실행` 단계를 강제한다.
- 미리보기에는 원본 경로, 변경 후 경로, 동작, 경고/충돌 상태를 표시한다.
- 오류나 차단 항목이 있으면 해당 항목의 이유를 표시하고, 위험한 상태에서는 실행 버튼을 비활성화한다.
- 작업 단위 JSON 로그를 로컬 앱 데이터 폴더에 기록한다.
- 직전 성공 작업은 앱에서 즉시 Undo할 수 있다. 이후 사용자가 대상 파일을 수정했거나 동일 경로를 점유한 경우 덮어쓰지 않고 충돌을 보고한다.
- 숨김/시스템/보호 파일, 재분석 지점(심볼릭 링크·정션), 권한 없는 경로는 기본적으로 제외한다.
- 원본/대상 폴더의 정규화된 실제 경로를 검사해 선택 범위 밖 변경과 경로 탈출을 막는다.
- 기존 파일을 자동 덮어쓰지 않는다.

### 2.4 배포와 운영

- 상용 라이브러리·유료 서비스를 사용하지 않는다.
- 로컬 서버 없이 실행한다.
- 단일 실행 파일을 우선 제공하고, 바로가기 및 휴대용 압축 파일도 만든다.
- 파일 정리/폴더 주제의 앱 아이콘과 원클릭 실행 바로가기를 제공한다.
- Windows 탐색기 우클릭 메뉴 등록/해제 토글을 제공한다.
- GitHub 저장소에 소스를 공개하고 GitHub Releases로 배포한다.
- 앱 안에서 새 릴리스를 확인하고, 사용자가 승인한 경우에만 다운로드한다. 다운로드한 배포물은 게시된 SHA-256 체크섬과 대조한다.
- 최종 배포물과 로그 예시에는 사용자명, 로컬 절대 경로, 테스트 파일명 등 개인정보를 포함하지 않는다.

## 3. 기술 방향

### 3.1 스택

- **언어:** Python 3.12+
- **GUI:** 표준 `tkinter`/`ttk`
- **파일 처리:** Python 표준 라이브러리(`pathlib`, `shutil`, `os`, `stat`)
- **Windows 연동:** 표준 `winreg`, 필요 최소 범위에서 `ctypes`
- **패키징:** PyInstaller(오픈소스, 부트로더 예외 조항으로 생성 실행 파일 배포 가능)
- **테스트:** `pytest` 및 임시 디렉터리 기반 파일 시스템 테스트
- **업데이트 확인:** 표준 `urllib`로 GitHub Releases API 조회; 앱 내 웹 서버 없음
- **CI/배포:** GitHub Actions의 무료 공개 저장소 실행 환경

`tkinter`/`ttk`는 추가 GUI 런타임 라이선스와 대형 의존성을 피하면서 PyInstaller 단일 실행 파일을 만들기 쉽다. UI는 운영체제 글꼴, 명확한 간격, 색상 토큰, 커스텀 ttk 스타일로 현대화한다. 구현 전 제시하는 세 HTML 후보 중 사용자가 선택한 시각 방향을 이 스택으로 재현한다.

### 3.2 아키텍처

파일 시스템 변경 계획과 GUI를 분리한다.

1. **Scanner**가 입력을 정규화하고 파일, 속성, 보호 상태를 읽는다.
2. **Planner**가 실제 파일을 건드리지 않고 변경 계획과 경고를 생성한다.
3. **Preview UI**가 계획을 표로 보여주고 확인을 받는다.
4. **Executor**가 계획의 사전 조건을 다시 검증한 뒤 단계적으로 적용한다.
5. **Journal**이 각 단계의 전후 상태를 원자적으로 기록한다.
6. **Undo Engine**이 직전 완료 작업의 역연산을 보수적으로 실행한다.

계획 항목에는 원본/대상 경로, 작업 종류, 예상 파일 크기·수정시각, 충돌 정책, 실행 상태가 포함된다. 실행 직전 파일 상태가 미리보기 시점과 달라졌으면 중단하고 재미리보기를 요구한다.

## 4. UX 설계와 추가 편의 기능

두 핵심 기능을 별도 앱처럼 나누지 않고 한 창의 작업 전환 구조로 통합한다. 공통 요소인 파일 선택, 규칙 설정, 미리보기, 실행, 로그, Undo는 같은 패턴을 사용해 학습 부담을 줄인다.

실제 사용자 흐름은 다음처럼 설계한다.

1. 앱 시작 화면에서 `폴더 평탄화` 또는 `순번 이름짓기`를 선택한다.
2. 탐색기에서 드래그 앤 드롭하거나 찾아보기로 입력을 고른다.
3. 자주 쓰는 안전 기본값이 미리 선택된 옵션을 조정한다.
4. `미리보기 만들기`를 누르면 변경 표와 요약(대상 수, 총 크기, 경고 수)을 확인한다.
5. 모든 차단 문제를 해결한 뒤에만 `적용` 버튼이 활성화된다.
6. 적용 결과와 로그 위치를 확인하고 필요하면 `직전 작업 실행 취소`를 누른다.

추가할 유용 기능은 다음과 같다.

- **파일 필터:** 확장자 포함/제외, 최소·최대 크기, 숨김 파일 제외
- **간편 프리셋:** 사진 모으기, 문서 모으기, 전체 파일 모으기; 사용자가 마지막 설정을 다시 사용
- **대상 폴더 열기:** 작업 완료 후 탐색기에서 결과 위치를 즉시 연다
- **미리보기 검색/필터:** 충돌·경고 항목만 빠르게 보기
- **계획 내보내기:** 실제 작업 전 변경 계획을 CSV로 저장해 검토 가능
- **디스크 공간 사전 검사:** 복사 예정 크기보다 여유 공간이 부족하면 실행 차단
- **작업 요약:** 성공/건너뜀/실패 수와 실패 사유를 개인정보가 제거된 형태로 표시
- **테마와 접근성:** 시스템 고대비/배율 대응, 키보드 탐색, 위험 동작의 색상 외 텍스트 구분

초기 버전에서는 핵심 안전성과 직결되는 필터, 디스크 검사, 결과 폴더 열기, 미리보기 검색을 포함한다. CSV 내보내기와 사용자 프리셋 저장은 핵심 기능 안정화 후 포함 여부를 재평가한다.

## 5. Windows 탐색기 우클릭 통합

기본 등록 위치는 현재 사용자 전용 `HKEY_CURRENT_USER\Software\Classes`이므로 관리자 권한이 필요하지 않다. 모든 사용자용 `HKEY_LOCAL_MACHINE` 등록은 별도 선택지로 제공할 수 있으나 관리자 승격이 필요하다.

- 폴더 우클릭: `FileFlow Lite로 이 폴더 평탄화`
- 폴더 배경 우클릭: `FileFlow Lite 순번 이름짓기`
- 앱 설정: `탐색기 메뉴 등록` / `탐색기 메뉴 해제`

Windows가 여러 선택 파일을 레지스트리 명령 하나에 안정적으로 전달하지 않는 제약을 피하기 위해, 순번 이름짓기 메뉴는 현재 폴더를 앱에서 열고 사용자가 파일을 체크하는 방식으로 시작한다. 탐색기 다중 선택을 정확히 전달하는 네이티브 셸 확장은 복잡도와 공격 표면이 커지므로 초기 버전 범위에서 제외한다.

등록 시 실행 파일 경로는 레지스트리 명령 인수로 안전하게 인용하고, 앱은 전달된 경로를 다시 정규화·검증한다. 해제는 FileFlow Lite가 만든 키만 정확히 삭제한다.

## 6. GitHub 배포와 업데이트

- 공개 GitHub 저장소를 사용해 소스, 이슈, 릴리스 노트를 관리한다.
- 태그가 생성되면 GitHub Actions가 테스트, PyInstaller 빌드, 악성코드/비밀정보 점검, SHA-256 체크섬 생성을 수행한다.
- 결과물은 단일 실행 파일, 휴대용 ZIP, 체크섬 파일로 GitHub Release에 첨부한다.
- 앱은 시작 시 자동 다운로드하지 않는다. 설정에 따라 새 버전 존재 여부만 확인하거나 사용자가 `업데이트 확인`을 눌렀을 때 조회한다.
- 업데이트 파일은 HTTPS로 내려받고 SHA-256을 확인한다. 검증 실패 시 삭제하고 설치를 차단한다.
- 실행 중인 단일 EXE를 직접 덮어쓰지 않고 별도 업데이트 헬퍼 또는 다음 시작 시 교체 방식을 사용한다. 최종 방식은 구현 전 위협 모델과 PyInstaller 동작을 검증해 결정한다.
- 저장소 URL과 게시 권한은 사용자가 UI 방향을 선택한 뒤 GitHub 업로드 단계에서 확인한다. 사용자의 명시적 승인 없이 원격 저장소 생성, 커밋 푸시, 릴리스 게시를 수행하지 않는다.

## 7. 구현 계획과 설계 결정 이유

1. **인계 문서와 UI 방향 확정**  
   자기완결형 설계를 먼저 남기고 세 HTML 후보를 실제 화면처럼 비교한다.
2. **핵심 모델·안전 정책 구현**  
   UI보다 먼저 순수 계획 생성기와 경로/보호 파일 검사를 테스트해 파괴적 오류를 줄인다.
3. **평탄화와 순번 이름짓기 계획기 구현**  
   모든 작업을 미리보기 가능한 불변 계획으로 표현한다.
4. **저널·실행·Undo 구현**  
   단계별 성공을 기록해 부분 실패 시에도 실제 상태를 추적한다. Undo는 무조건 덮어쓰기보다 안전한 중단을 우선한다.
5. **선택된 GUI 구현**  
   공통 흐름과 접근성, 대량 파일에서도 반응성을 유지하는 백그라운드 스캔을 적용한다. 실제 파일 적용은 단일 제어 경로로 직렬화한다.
6. **탐색기 통합·바로가기·아이콘 구현**  
   기본은 관리자 권한 없는 현재 사용자 등록이다.
7. **업데이트·패키징·보안 점검**  
   다운로드 검증, 의존성 고정, 비밀정보 검색, 테스트, 깨끗한 환경 실행을 확인한다.
8. **GitHub 공개 및 릴리스**  
   사용자가 지정하거나 승인한 저장소에 업로드하고 릴리스 기반 업데이트를 연결한다.

## 8. 제약 조건과 비목표

- Windows 10/11을 대상으로 하며 다른 운영체제는 초기 지원 대상이 아니다.
- 네트워크 드라이브와 클라우드 동기화 폴더는 동작 가능성이 있지만, 원자성·지연·동시 수정 위험을 경고하고 초기 테스트 범위를 별도로 둔다.
- NTFS 권한, ADS, 파일 소유권을 완전 복제하는 백업 도구가 아니다.
- 휴지통 기반 복구를 Undo의 유일한 수단으로 사용하지 않는다.
- 시스템 보호 영역을 강제로 수정하거나 권한을 우회하지 않는다.
- 자동 업데이트는 무인·강제 설치가 아니며 사용자 승인과 무결성 확인을 거친다.
- 레지스트리 기반 우클릭 통합은 현재 사용자 범위를 기본으로 한다.
- UI 후보는 설계 승인용 정적 HTML이며 실제 파일 작업을 수행하지 않는다.

## 9. 보안 및 개인정보 기준

- 로그에는 필요한 로컬 경로가 포함될 수 있으므로 기본 저장 위치를 사용자 프로필의 앱 전용 폴더로 제한하고 외부 전송하지 않는다.
- 진단 정보 내보내기 기능을 추가할 경우 사용자명과 경로를 마스킹한다.
- 심볼릭 링크·정션을 따라가지 않고, 각 실행 단계에서 경로 경계를 재검증한다.
- TOCTOU 위험을 줄이기 위해 미리보기 때 기록한 크기·수정시각·파일 식별 정보를 실행 직전에 비교한다.
- 업데이트는 공식 GitHub 저장소의 고정된 HTTPS 엔드포인트만 사용하고 체크섬을 검증한다.
- 레지스트리와 명령행 인수는 셸 문자열 결합을 최소화하고 경로 인용 및 입력 검증을 적용한다.
- 압축 파일 생성 전 `.git`, 캐시, 로컬 로그, 테스트 산출물, 환경 파일, 사용자 경로가 포함되지 않았는지 검사한다.

## 10. 현재 상태

**v0.1.0 구현·검증·패키징·GitHub 공개 및 원격 CI 재검증 완료.**

- 배포 패키지 폴더 `FileFlow-Lite`를 생성했다.
- 요구사항, 안전 정책, 기술 방향, UX 흐름, 탐색기 통합, GitHub 업데이트 방향을 이 문서에 정리했다.
- 서로 다른 UI 스타일 후보 3개를 `ui-candidates` 폴더에 독립형 HTML로 작성했다.
  - 후보 A: 차분한 대시보드형 — 기능 전환과 옵션, 미리보기를 한 화면에 균형 있게 배치
  - 후보 B: 탐색기 친화형 — Windows 파일 탐색기와 유사한 밀도 높은 목록 중심 화면
  - 후보 C: 단계별 안내형 — 초보 사용자와 파괴적 작업 방지에 유리한 4단계 마법사 흐름
- 인앱 브라우저의 로컬 `file://` 접근이 실행 환경의 보안 정책으로 차단되어 자동 렌더링 검증은 수행하지 못했다. 로컬 서버를 띄우지 않는다는 제품 제약과 브라우저 보안 정책을 존중해 우회하지 않았다.
- 세 HTML 파일의 문서 구조를 파싱했으며 모두 제목과 유효한 태그 구조를 포함한다. 외부 URL 리소스는 0건이고 사용자명, `Users\\`, `AppData`, `OneDrive` 등 개인정보성 경로 표식도 0건이다.
- 사용자가 후보 A의 대시보드 구조와 후보 C의 단계별 최종 확인 흐름을 혼합하도록 선택했다.
- 구현은 A의 좌측 기능 탐색과 옵션·미리보기 동시 배치를 기본으로 하며, 실제 적용 직전에 C의 별도 최종 확인 단계를 강제한다.
- Python/Tkinter 핵심 앱, 평탄화, 순번 이름짓기, 미리보기, 최종 확인, JSON 로그, Undo를 구현했다.
- 평탄화 부분 실패 자동 롤백과 Undo 전체 사전검사를 추가해 부분 적용 위험을 줄였다.
- 현재 사용자 범위 탐색기 우클릭 메뉴 등록/해제와 GitHub Releases 업데이트 확인·SHA-256 검증을 구현했다.
- 파일 정리 주제의 아이콘, PyInstaller 단일 EXE, 휴대용 ZIP, 바탕 화면 바로가기 생성 스크립트를 구현했다.
- 자동화 테스트 17개가 통과했고 Bandit 정적 보안 분석에서 미해결 항목이 없다.
- 최종 EXE 시작 스모크 테스트를 통과했다.
- 배포 ZIP은 `FileFlow-Lite.exe`, `README.md`, `LICENSE`만 포함한다. 사용자명·로컬 경로·비밀정보 패턴이 검출되지 않았고 게시된 SHA-256이 실제 파일과 일치한다.
- 현재 PC의 바탕 화면에 `FileFlow Lite.lnk`를 만들고, HKCU 범위의 우클릭 메뉴 3개를 등록했다. 관리자 권한은 사용하지 않았다.
- 공개 소스 저장소: `https://github.com/jykim5215/fileflow-lite`
- v0.1.0 릴리스: `https://github.com/jykim5215/fileflow-lite/releases/tag/v0.1.0`
- 릴리스 자산으로 단일 EXE, 휴대용 ZIP, `SHA256SUMS.txt`를 게시했다.
- 앱의 업데이트 확인 엔드포인트가 위 저장소를 가리키며 현재 v0.1.0이 최신 버전으로 정상 인식된다.
- 최초 GitHub Actions 실행에서 테스트와 EXE 빌드는 통과했으나, Windows PowerShell 5.1이 BOM 없는 UTF-8 한글 문자열을 잘못 해석해 패키징 단계가 실패했다. 파일 처리 코드나 로컬 배포물의 실패가 아니라 CI 스크립트 인코딩 호환 문제였다.
- 사용자가 수정 진행을 승인했다. `scripts/*.ps1`의 실행 문자열을 ASCII 안전 형태로 정리했으며 비 ASCII 문자열 0건을 확인했다. Windows PowerShell 5.1 로컬 패키징, 자동화 테스트 17개, Bandit 정적 분석을 다시 통과했다.
- 수정 커밋 `33e3965`를 GitHub에 푸시했다. GitHub Actions 재실행 `29180737018`에서 테스트, EXE 빌드, 패키징, 체크섬 생성, 워크플로 아티팩트 업로드가 모두 성공했다.

## 11. 확정된 운영 결정

- UI는 A 대시보드와 C 단계별 최종 확인을 혼합한다.
- 표시 이름은 `FileFlow Lite`를 유지한다.
- 저장소는 인증된 GitHub 계정의 공개 저장소 `jykim5215/fileflow-lite`를 사용한다.
- 업데이트는 앱 시작 시 자동 조회하지 않고 사용자가 `업데이트 확인` 버튼을 눌렀을 때만 확인한다.
- 앱은 업데이트 ZIP을 자동 설치하거나 현재 EXE에 덮어쓰지 않는다. SHA-256 검증 후 폴더를 열어 사용자가 교체한다.
